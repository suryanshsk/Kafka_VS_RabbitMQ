RabbitMQ — Push
Broker decides delivery
Broker maintains connection & pushes messages instantly. Consumer must be ready. Risk: overwhelm slow consumers. Use prefetch count to limit.
Kafka — Pull
Consumer decides when
Consumer polls at its own pace. Can batch-fetch. Never overwhelmed. Consumer tracks its own offset. Scale independently per group.

Destructive Consumption
One time, then gone
RabbitMQ deletes after ACK. Only one consumer reads each message. Great for task queues, payments, emails — where you don't want double processing.
Message Retention
Stored, replayable
Kafka keeps messages (default 7 days, configurable forever). Multiple consumer groups read same data independently. Rewind & replay anytime.

Smart Broker vs Dumb Broker

Where does the intelligence live?

Smart Broker, Dumb Consumer
The broker is intelligent. It knows:
✅ Which consumer gets which message
✅ Routing rules (Exchange types)
✅ Whether message was delivered (ACK)
✅ Retry logic, dead letter queues
✅ Priority queues
✅ Tracks consumer state
Consumer is simple — just receives & ACKs. Broker does the heavy lifting.
Dumb Broker, Smart Consumer
The consumer is intelligent. It knows:
✅ Which offset to read from
✅ When to commit offsets
✅ How fast to consume
✅ How to rebalance in a group
✅ Whether to replay old data
✅ Its own position in the log
Kafka broker = dumb log store. Consumers do all the state management.
AspectRabbitMQ (Smart Broker)Kafka (Dumb Broker)
Routing logicIn the broker (Exchange)In the consumer (read from partition)
Consumer trackingBroker tracks delivery stateConsumer tracks offset
Failure recoveryBroker retries, dead-letterConsumer re-reads from offset
ScalingVertical (broker gets bigger)Horizontal (add partitions)
BottleneckBroker can become bottleneckLog is horizontally distributed
Why this matters for interviews: Kafka's "dumb broker" design is why it scales to millions of msgs/sec. The broker just appends to a log — no routing overhead. Consumer groups independently manage their own position.

When to Use What

Real-world examples — click a card to expand

Use RabbitMQ when...
  • Task queues (background jobs)
  • Email / SMS notifications
  • Microservice RPC calls
  • Complex routing needed
  • Per-message ACK & retry
  • Low latency responses
  • One consumer per message
Use Kafka when...
  • Event streaming / real-time data
  • Logs, analytics, tracking
  • Multiple consumers same data
  • Audit trail needed (replay)
  • High throughput (millions/sec)
  • Event sourcing / CQRS
  • Data pipeline / ETL
